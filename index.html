Phase 1: Efficient Detection

Leveraging battle-tested classic ML models like Isolation Forests and DBSCAN, we can flag outliers in massive structured datasets with incredible speed and accuracy. This deterministic approach forms a reliable, cost-effective backbone for detection.

Phase 2: Contextual RCA

This is where the magic happens. An LLM-powered agent takes the flagged anomaly and asks "Why did this happen?". Using Retrieval-Augmented Generation (RAG), it grounds its reasoning in your organization's knowledge bases (like Confluence) to provide a clear, actionable explanation.

Explainable Context-Aware Actionable

Solution in Action: The ADI Use Case

Anomalies aren't generic, so their analysis shouldn't be either. Here's how our system provides personalized reasoning during a data reconciliation task.

1

Anomaly Detected

During a routine pipeline run, the system identifies a significant data discrepancy between SAP and ADW tables.

2

Agent Queries Knowledge Base

The LLM agent is triggered. It queries the Confluence knowledge base via our custom MCP server, searching for relevant documentation, past incident reports, or SOPs related to this data pipeline.

3

Contextual Reason is Generated

The agent uses RAG to synthesize the information. It finds a recently updated document detailing a change in the SAP source system's schema that wasn't propagated to the ADW ingestion logic.

4

Actionable Insight Delivered

Instead of a vague alert, the on-call engineer receives a precise notification: "Anomaly in SAP-ADW sync likely caused by recent schema change (see Confluence doc #123). Suggested fix: Update ADW mapping."
